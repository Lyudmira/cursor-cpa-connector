# Claude Prompt Caching Fix (End-to-End) and Wildcard Claude Model Routing

## Commit

`4b37860` — `Fix Claude prompt caching end-to-end and extend to any Claude model`

## Symptom

Claude models (`claude-sonnet-5` etc.) routed through Cursor → Bifrost → muskapi never
showed a cache hit. `cached_read_tokens` was 0 on every request, including the second of
two back-to-back requests with an identical stable tool list — the scenario that should be
the easiest possible case for Anthropic prompt caching to succeed. muskapi's own prompt
caching was independently confirmed to work correctly (this session's own Claude Code
traffic, routed through the same `ANTHROPIC_BASE_URL`/`ANTHROPIC_API_KEY`, has always shown
non-zero cache reads), which pointed at something specific to the Cursor → Bifrost → muskapi
path rather than a muskapi-side defect.

This turned into a five-layer investigation. Every layer independently produced a real,
reproducible zero-cache-hit result; fixing one layer only exposed the next one underneath.
Each is described below in the order it was found, including the dead ends, because the
dead ends are exactly what make the eventual root causes verifiable rather than assumed.

## Investigation and fix, layer by layer

### Layer 1 — a prior connector patch (`4a83267`) added `cache_control`, but Bifrost's own OpenAI request marshaler silently stripped it

The connector already had a patch (`4a83267`, "Restore Claude tool-prefix cache
compatibility") that injected `cache_control: {"type":"ephemeral"}` on the last tool of a
Cursor request, and preserved it through Bifrost's Responses↔Chat struct conversions. Its
own tests passed. But those tests only asserted on intermediate Go structs
(`ResponsesTool`/`ChatTool`), never on the bytes actually sent to the upstream.

Reading `core/providers/openai/types.go`'s `OpenAIChatRequest.MarshalJSON` found the real
choke point:

```go
// OpenRouter forwards Anthropic-style cache_control breakpoints to the
// underlying model, so we must preserve cache_control ... when targeting OpenRouter.
keepCacheControl := req.Provider == schemas.OpenRouter
```

Every `ChatTool`/message `cache_control` field was stripped before serialization unless the
Bifrost provider was literally `openrouter`. muskapi, configured as a custom provider with
`base_provider_type: "openai"`, was never going to satisfy that check — and
`SupportedBaseProviders` for custom providers doesn't even include `OpenRouter` as an
option, so no custom provider ever could. The marker added by the connector's Cursor parser
was faithfully carried through every internal struct conversion and then deleted one layer
downstream, right before the request left Bifrost.

**Dead end explored and ruled out first:** an experimental build with the OpenRouter-only
check commented out (`keepCacheControl` forced `true`) confirmed `cache_control` now reached
muskapi on the wire — and `cached_read_tokens` was *still* 0. This ruled out "the field never
left Bifrost" as the sole explanation and pointed at something further downstream.

### Layer 2 — muskapi's OpenAI-Chat-Completions surface doesn't honor `cache_control`; its native Anthropic Messages endpoint does

`cache_control` is not part of the real OpenAI Chat Completions spec — it's an
Anthropic-specific extension. Bifrost was talking to muskapi via
`core/providers/openai`, i.e. treating it as a generic OpenAI-compatible backend and relying
on muskapi's own OpenAI-compat shim to notice and forward the non-standard field into its
internal Anthropic call. muskapi also exposes a native Anthropic Messages endpoint (the same
one Claude Code itself talks to via `ANTHROPIC_BASE_URL`/`ANTHROPIC_API_KEY`), which Cursor
cannot be pointed at directly — Cursor's custom-model configuration only accepts an
OpenAI-compatible base URL.

The fix does not require Cursor to talk to muskapi's Anthropic endpoint directly. Cursor
only ever talks to Bifrost's single OpenAI-compatible surface; what Bifrost does on the
*outbound* leg to muskapi is a config choice, invisible to Cursor. `muskapi-anthropic` was
registered as a second Bifrost provider (`base_provider_type: anthropic`, `base_url:
https://api.muskapi.cc` with no `/v1` suffix), and Claude models were routed to it instead
of the OpenAI-compatible `muskapi` provider. This runs Bifrost's own `core/providers/anthropic`
code path, which already builds protocol-correct Anthropic requests (correct `x-api-key`
header, `anthropic-version`, and `cache_control` field placement) with none of the Layer 1
stripping logic in its path.

Authentication was verified directly against muskapi before wiring this in: both `x-api-key`
and `Authorization: Bearer` worked against `https://api.muskapi.cc/v1/messages` with the
plain `ANTHROPIC_API_KEY` value, so no extra header translation was needed.

**Dead end explored and ruled out:** after wiring up the native-Anthropic route, the very
next synthetic test still showed 0 cache activity. This turned out to be a test-methodology
problem, not a real regression — see Layer 3.

### Layer 3 — a synthetic Chat-Completions-shaped test never triggers the connector's own cache-injection logic

`addCursorClaudeToolCacheBreakpoint` is wired into the **Responses API** Cursor parser only.
A hand-built `POST /cursor/v1/chat/completions` request in generic OpenAI Chat Completions
shape doesn't go through that parser, so a synthetic test built that way never carries
`cache_control` at all — the resulting "0 cache hit" was a correct, expected result for a
request with no cache marker in it, not evidence that anything downstream was broken. This
was confirmed by capturing the actual wire body via a transparent MITM relay pointed
temporarily at a local echo server (mirroring the request to a local capture file while still
forwarding it to the real upstream so production traffic wasn't disrupted) and finding zero
`cache_control` occurrences anywhere in it.

A minimal, curl-only reproduction directly against `https://api.muskapi.cc/v1/messages`
(bypassing Bifrost and its Cursor parser entirely) with an explicit, hand-added
`cache_control` on the last tool was used from this point on to isolate "does muskapi honor
this marker" from "does Bifrost's Cursor parser correctly add it."

### Layer 4 — `cache_control` on a tool alone produces zero cache activity on muskapi; it needs to be on a message content block too

With `cache_control` explicitly present only on the last tool (and nothing else), two
identical direct requests to muskapi's native endpoint produced:

```
Request 1: cache_creation_input_tokens: 0, cache_read_input_tokens: 0
Request 2: cache_creation_input_tokens: 0, cache_read_input_tokens: 0
```

Not just zero reads — zero writes too. No cache checkpoint was ever created. Adding a second
`cache_control` on the last content block of the `system` message (keeping the tool-level one
too) produced the expected result:

```
Request 1: cache_creation_input_tokens: 29513, cache_read_input_tokens: 0
Request 2: cache_creation_input_tokens: 0,     cache_read_input_tokens: 29513
```

A follow-up test with `cache_control` **only** on the message (no tool breakpoint at all)
also cached correctly and read back the full token count including the tools, confirming
Anthropic's breakpoints cover everything earlier in the request's fixed content ordering
(tools, then system, then messages) — a message-level breakpoint already subsumes the tools.
The tool-level breakpoint is therefore not just insufficient on its own, it's unnecessary
once any message-level breakpoint is present. `addCursorClaudeToolCacheBreakpoint` was
changed to never mark a tool directly; every breakpoint is message-level from this point on
(freeing up breakpoint budget spent later in Layer 5).

The connector's Cursor parser also assumed a dedicated `system`/`developer` message would
always be present to anchor this breakpoint on. Capturing real Cursor traffic (via the same
MITM relay technique) showed this assumption is false: production Cursor requests carry no
`system` field and no `role: "system"` message at all — all context, including what would
normally be a system prompt, is folded into the first message, which is `role: "user"`. The
stable-anchor breakpoint now falls back to the first message in the array when no
system/developer message is present, and this first message was independently confirmed to
stay byte-identical across a whole real conversation while later turns are appended after it.

**Dead end explored and ruled out:** the model name Cursor actually sends is
`muskapi/claude-sonnet-5` (Bifrost's own `provider/model` addressing convention, matching how
this kit's own `init_bifrost_config.py` registers Claude models), not the bare
`claude-sonnet-5` the original `strings.HasPrefix(model, "claude-")` check expected. This
made every real Cursor request silently no-op the entire cache-injection function regardless
of any other logic error. Found via a one-line temporary debug log printing `req.Model` on a
real Cursor request, which showed the prefixed form directly. Fixed with a small
`isClaudeCursorModel` helper that strips an optional `provider/` prefix before matching.

### Layer 5 — a single trailing breakpoint (even two, by raw index) doesn't let the cache grow with the conversation

After Layers 1–4, isolated back-to-back identical requests cached correctly, and even a
3-turn growing conversation showed a promising pattern once retested with adequate spacing
between requests (Anthropic ephemeral cache writes take a nontrivial, variable amount of
time — observed empirically at roughly 15–30s in this environment — to become read-visible;
requests fired only a few seconds apart can miss a cache written moments earlier purely from
this propagation lag, independent of any placement logic). But real, sustained Cursor usage
showed a different, deterministic failure: `cached_read_tokens` stayed pinned at whatever the
very first follow-up turn happened to cache, and every turn after that did a full-price
rewrite — not intermittently, consistently.

The breakpoint (at this point: stable anchor + the single latest message) only ever moved to
"the last message." On turn *N* it lands at index `n-1`; turn *N−1*'s own breakpoint had been
at *its* last message, which by turn *N* has been pushed back to an earlier index — by
exactly 2 positions for a plain user/assistant exchange (Cursor appends the assistant's reply
plus a new user query), by more when a turn produces several tool-call/tool-result items.
Marking "the last two messages by raw index" was tried next and still failed the same way,
because the raw gap between turns isn't fixed at 2 — a tool-calling loop within one turn
can insert an arbitrary number of items before the next real user message.

The fix that actually holds regardless of how many items a turn produces: track the last two
messages **by role**, not raw position. Cursor's history is a strict append — earlier
messages are never mutated. Whatever user message was "the latest" when the previous turn
built its own breakpoint is therefore always exactly "the second-to-last user message" by the
time the current turn looks at the same history, no matter how much non-user content (tool
calls, tool results, assistant text) sits between them. This was verified directly: a
synthetic turn-2/turn-3 pair with a 2-item gap between them showed the position that mattered
(turn 2's own latest message) get correctly re-marked in turn 3 as its second-to-last
user-role message.

A follow-up real-usage report surfaced one more gap this didn't cover: caching worked across
user turns, but a Cursor tool-calling loop **within** a single turn (assistant tool_call →
tool_result → tool_call → ...) still showed uncached full rewrites on every round, until the
turn ended. The user-role tracking above only moves when a new *user* message appears — it
never advances during a run of tool_call/tool_result items that all belong to the same turn's
still-open loop, so the growing tail inside that loop was never itself getting cached.

The final scheme spends all 4 of Anthropic's allowed breakpoints per request, verified
empirically that a message-only breakpoint subsumes the need for a separate tool-level one
(Layer 4), which freed the budget for this:

1. **Stable anchor** — system/developer message, or the first message as fallback.
2. **Absolute last message**, any role — lets a tool-calling loop *within* one turn
   accumulate cache: each round appends to the end of the array, and marking the true last
   item every request lets the next round in the same loop read what the previous one wrote
   moments earlier.
3. **Absolute second-to-last message** — pairs with #2 for the common case where a turn
   produces exactly one new item since the previous request in the same loop; both positions
   must be present for the read to hit turn-over-turn.
4. **Second-to-last *user*-role message** — realigns across a full turn boundary, where more
   than the 1–2 items #2/#3 assume get appended at once.

This was verified two ways. A unit test simulates the exact sequential single-item-per-call
growth of a real tool-calling loop and asserts, at every step, that the previous call's
absolute-last position is exactly the current call's absolute-second-to-last position (the
alignment invariant the whole scheme depends on). Against production traffic, four
consecutive real turns with adequate spacing showed:

```
Turn 1: write 22596, read 0        (first write, expected)
Turn 2: read 22596,  write 15      (exactly Turn 1's total, small delta written)
Turn 3: read 22611,  write 16      (exactly Turn 2's total, small delta written)
Turn 4: read 22627,  write 20      (exactly Turn 3's total, small delta written)
```

each turn reading exactly what the previous turn wrote and only paying for the small
per-turn delta — the cache accumulating with the conversation instead of resetting.

## Extending Claude routing to any model, not an enumerated list

Layer 2's fix originally routed only the three Claude model names already known to
`init_bifrost_config.py` (`claude-sonnet-5`, `claude-fable-5`, `claude-opus-4-8`) to the new
`muskapi-anthropic` provider — a new Claude model (e.g. `claude-haiku-4-5-20251001`, used to
validate this) would have silently fallen through to the broken OpenAI-compatible path, or
failed outright.

Two things were needed, both verified live before being folded into `init_bifrost_config.py`:

- The routing rule became a single wildcard CEL expression, `model.startsWith('claude-')`
  (`ValidateCELExpression` in Bifrost's own routing package explicitly allows
  `.startsWith(`), replacing the three exact-match rules. `routing_targets.model` is left
  `NULL` — Bifrost's router resolves a `NULL` target model as "forward the client-requested
  model unchanged" (see `Model *string ... // nil = use incoming model` in
  `framework/configstore/tables/routingrules.go`), so any current or future Claude model name
  routes correctly with no code change.
- The `muskapi-anthropic` provider key's model allowlist had to change from the three
  hardcoded names to `["*"]`. Bifrost keys deny-by-default on an empty model list
  (`key.Models is empty → exclude key`, per `core/bifrost.go`); a routing match alone is not
  sufficient; the key itself must also declare it's allowed to serve the model.

Verified against `claude-haiku-4-5-20251001`, a model that had never been listed anywhere in
this kit's configuration: it routed to `muskapi-anthropic` and completed successfully on the
first try after the wildcard rule and `["*"]` allowlist were applied.

One general operational note surfaced repeatedly throughout this investigation and is
unrelated to the fix itself: Bifrost's config admin API (`PUT /api/providers/...`,
`PUT /api/providers/{provider}/keys/{key_id}`) is the durable way to change provider/key
config on a running instance. Editing `config_providers`/`config_keys` rows directly via SQL
against the live `config.db` was observed to be silently reverted by Bifrost's own
config-reconciliation logic on the next restart in this environment; going through the admin
API instead was confirmed to persist correctly across restarts.

## Fix summary

- `core/providers/openai/types.go`'s `OpenAIChatRequest.MarshalJSON` `cache_control`
  stripping was not patched (it's correct, documented OpenRouter-specific behavior);
  instead, Claude traffic was routed around it entirely via a second, native-Anthropic
  Bifrost provider.
- `transports/bifrost-http/integrations/cursor.go`:
  - `isClaudeCursorModel` — model-prefix matching that tolerates a `provider/model` address.
  - `addCursorClaudeToolCacheBreakpoint` — no longer marks a tool directly; delegates
    entirely to the message-level scheme below.
  - `setCursorMessageCacheBreakpoint` — marks a single message correctly regardless of
    whether it's a regular content message (content-block-level `cache_control`) or a
    `function_call`/`function_call_output` item (message-level `cache_control`).
  - `addCursorClaudeSystemCacheBreakpoint` — the 4-breakpoint scheme: stable anchor,
    absolute last message, absolute second-to-last message, second-to-last user-role
    message.
- `install-cursor-cpa-bifrost-patches.ps1` — the Bifrost patch stage now installs this final
  version directly, upgrading idempotently from either of the two earlier generations of this
  function family that may already be installed.
- `init_bifrost_config.py` — registers the `muskapi-anthropic` provider
  (`base_provider_type: anthropic`, derived from the primary URL with the `/v1` suffix
  stripped), a `["*"]`-allowlisted key, and the single wildcard routing rule; Claude models
  were removed from `PRIMARY_MODELS` (which still routes everything else to the
  OpenAI-compatible `muskapi` provider) since they're now handled entirely by the wildcard
  rule.
- `bifrost_cursor_compat_test.go` — covers model-prefix matching (including the
  `provider/model` form), the system-message/first-message stable-anchor fallback, explicit
  client-supplied `cache_control` suppressing auto-injection, and — the regression that
  mattered most — that the breakpoint window realigns correctly both across a raw
  turn-boundary sequence with a 2-item gap and across a step-by-step simulated tool-calling
  loop with a 1-item-per-call gap.

## Verification

- `go test ./core/schemas -run CursorConnector` and
  `go test ./transports/bifrost-http/integrations -run CursorConnector` both pass.
- `install-cursor-cpa-bifrost-patches.ps1`'s `Apply-Bifrost-Patch` function was run end to
  end against a fresh, unpatched clone of the Bifrost source (not the working checkout used
  during live debugging): the cascading patch stages applied cleanly, `go test` passed, and
  `docker build` produced a working image — confirming the install script reproduces the fix
  from scratch, not just the manually-edited checkout used to develop it.
- Direct requests against `https://api.muskapi.cc/v1/messages` isolated and confirmed each of
  Layers 1, 3, and 4 independently of Bifrost, ruling out coincidental interaction effects.
- Real Cursor traffic (`User-Agent: Cursor/1.0`, captured via a transparent MITM relay that
  forwarded to the real upstream so production usage was never disrupted) confirmed the final
  request shape (no system message, `provider/model`-prefixed model name, growing message
  history) matches what the unit tests and the wildcard-routing change assume.
- Four consecutive real production turns showed monotonically accumulating
  `cached_read_tokens` exactly matching each prior turn's write, as quoted in Layer 5 above.
- `claude-haiku-4-5-20251001` — never previously listed in any config in this kit — routed
  and completed successfully immediately after the wildcard routing change, with no code
  change and no restart-losing-config issue once applied through the admin API.
