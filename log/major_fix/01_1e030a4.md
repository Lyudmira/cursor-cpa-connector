# Claude Responses Tool Continuation Fix

## Commit

`1e030a4` — `Normalize Cursor tool outputs for Claude Responses`

## Symptom

Cursor could start a `claude-sonnet-5` conversation through Bifrost and muskapi, and Claude could request a local tool successfully. After Cursor executed the tool, the follow-up request failed immediately with HTTP 502 while Cursor displayed `planning next move`. Plain text conversations continued to work.

## Investigation

Bifrost logs showed that the failing requests were routed directly to `muskapi/claude-sonnet-5` as `responses_stream`; CPA and the backup provider were not involved. The failure appeared only after the input history contained `function_call` and `function_call_output` items.

Historical logs contained successful Claude conversations with the same non-message `role` fields, so removing `role` was tested independently. A real failed request was reconstructed from the Bifrost logging database and replayed through the local Cursor-compatible endpoint.

The controlled results were:

1. A minimal text-only request returned HTTP 200 and exactly `ok`.
2. Removing `role` from the `function_call` item still returned HTTP 502.
3. Keeping the same request but changing the tool output block from `type: "text"` to `type: "input_text"` returned HTTP 200, and Claude correctly continued with `CLAUDE-C01-TOOL-TURN-OK`.

This isolated the compatibility failure to Cursor's non-standard tool-result content alias:

```json
{
  "type": "function_call_output",
  "output": [
    {
      "type": "text",
      "text": "..."
    }
  ]
}
```

The current muskapi Claude Responses adapter requires the standard input-history content type:

```json
{
  "type": "function_call_output",
  "output": [
    {
      "type": "input_text",
      "text": "..."
    }
  ]
}
```

## Fix

The Bifrost installation patch now updates `transports/bifrost-http/integrations/cursor.go` and inserts `normalizeCursorFunctionCallOutputs` into both Cursor request parsing paths. For every `function_call_output`, it rewrites:

- `text` to `input_text`
- `output_text` to `input_text`

The normalization runs after Cursor tool-result merging and before general input content normalization. It is intentionally limited to Cursor's `function_call_output` history and does not apply Codex-specific request behavior to Claude or other providers.

The installer remains idempotent by checking for the helper function before modifying the source. It also runs `gofmt` on the patched Cursor integration file before building the local Bifrost image.

## Verification

The modified Bifrost image built successfully and was restarted as the `bifrost` container on `127.0.0.1:8081`. A real Cursor Claude tool call then completed successfully after restart, confirming that the production request path now normalizes the tool-result block before provider forwarding.

The standalone transport integration test command could not compile against the repository's independently versioned Go modules because the transport module resolved released `core`, `framework`, and plugin versions that did not match the checked-out development source. The full Docker build uses a generated Go workspace containing the checked-out local modules and completed successfully, which validates the patched source in the same dependency topology used by the deployed image.
